# Other — Research System Metadata
This file contains supporting structure for the SaaS GTM Intelligence Repository.

It defines how research is processed, categorized, and interpreted across all expert sources.

## Purpose of This Repository
This system is designed to transform raw LinkedIn content into structured GTM intelligence.
It is not:
- A content summary archive
- A personal opinion log
- A marketing inspiration board
It is:
- A structured research system for understanding modern SaaS growth dynamics.

## Research Methodology
Each source is processed using 3 layers:

### 1. Raw Signal Extraction
- Identify core claim or insight
- Remove storytelling noise
- Capture original intent

### 2. Concept Tagging
Each insight is categorized into system-level tags:

Examples:
- trust
- distribution
- AI automation
- GTM systems
- product-led growth
- demand generation
- content strategy
- customer intelligence

### 3. System Interpretation
Each insight is mapped into one of four macro systems:

#### 1. Trust System
How buyers validate and believe information

#### 2. Intelligence System
How companies understand customers and markets

#### 3. Distribution System
How attention and visibility are generated

#### 4. Execution System
How work is actually produced (manual vs AI-driven)

## Expert Classification Logic
Each expert in the system is evaluated based on:
- Primary domain (e.g. PLG, demand gen, SEO, AI systems)
- Level of operational experience (founder, operator, advisor)
- Type of thinking:
  - Tactical (execution-focused)
  - Strategic (system-focused)
  - Structural (org or market design)

## Output Structure Philosophy
The goal of this system is not to store information.

It is to answer:
- “What actually drives SaaS growth in the AI era?”

Everything in the repository should eventually map to:
### Growth Equation
- Trust × Intelligence × Distribution × Speed-to-Value

## Data Processing Rules
When adding new content:
1. Do not store content without tagging
2. Every insight must map to at least one system layer
3. Duplicate ideas across experts should be merged into a single concept node
4. Contradictions between experts should be preserved (not removed)

## Contradiction Policy
Conflicting views are valuable.
Examples:
- “AI increases content value” vs “AI destroys content differentiation”
- “Funnels still work” vs “Funnels are obsolete”
- “Execution matters” vs “Execution is commoditized”

These contradictions are treated as:
- signal of system evolution, not error

## Future Extensions
Planned additions to this system:
- `knowledge_graph.md` → relationship mapping between experts
- `contradictions.md` → structured disagreements between frameworks
- `clusters.md` → emergent GTM categories
- `system_model.md` → final SaaS GTM operating model

## Core Principle
This system assumes:
- Modern SaaS growth is not linear.  
- It is a multi-layer feedback system of trust, intelligence, distribution, and execution loops.
